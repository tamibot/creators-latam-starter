03 · Ventana de contexto

La regla del medio millón.

Antes teníamos 124.000 tokens. Hoy casi todos los modelos llegan a 1 millón. Suena a mucho, pero colapsa más rápido de lo que creés.

Cuando la conversación está llegando a los ~500k tokens, compilá. Todos los modelos tienen la opción: resume toda la conversación y reiniciás desde cero, conservando el hilo. No es un riesgo, es higiene. Si no compilás, vas a empezar a generar errores del mismo modelo: responde raro, se olvida del contexto real, confunde archivos.

Compilar no es admitir derrota. Es la buena práctica más barata que existe.
04 · Arquitectura de carpetas

Donde vive todo. Sin excepciones.

Si usás Mac, arranca en el escritorio — iCloud respalda todo automáticamente. Una carpeta madre, y dentro: una subcarpeta por cliente. Tu propia agencia es un cliente más.

carpeta · cliente · proyecto
📁 Escritorio/
└── 📁 Cliente-X/            # una carpeta por cliente
    ├── 📁 Proyecto-A/
    │   ├── README.md           # descripción del proyecto
    │   ├── CLAUDE.md           # reglas de oro que el modelo lee auto
    │   ├── LLM.md              # manual de bienvenida para agentes
    │   ├── .env                # credenciales locales · NUNCA subir
    │   ├── 📁 .claude/agents/  # 14 agentes preconfigurados
    │   ├── 📁 .claude/skills/  # PDF→MD, Video→texto, Plan paso a paso
    │   ├── .claude/settings.json   # permisos preaprobados
    │   ├── 📁 tooling/          # install.sh + docs de herramientas
    │   ├── 📁 templates/       # NV, brief, API research
    │   ├── 📁 documentation/   # todo lo del cliente
    │   ├── 📁 workflows/       # flujos específicos del proyecto
    │   ├── 📁 output/          # entregables finales — lo que el cliente ve
    │   └── 📁 data_original/   # PDFs, screenshots, raw uploads
    ├── 📁 Proyecto-B/
    └── 📁 Proyecto-C/

¿Para qué sirve cada carpeta?

  • README.mdLo primero que lee cualquier persona (o agente) que entra a la carpeta.
  • .envCredenciales. Queda local, nunca se sube al repo.
  • documentation/Toda la documentación que el cliente te pasa, ordenada.
  • workflows/Flujos de trabajo específicos del proyecto, como receta.
  • output/Lo único que le importa al cliente. Cero fricción para encontrarlo.
  • data_original/El buzón de entrada. PDFs, contratos, screenshots. Sin tocar.

06 · Skills fundacionales

Las tres habilidades que siempre instalamos.

Los skills son habilidades que le enseñás al sistema paso a paso. Hay tres que montamos por defecto en todo proyecto.

Skill 01

PDF / Docs a Markdown

Los modelos leen markdown 10× más rápido y barato que PDFs. Usa marker para PDFs técnicos y markitdown para todo lo demás (DOCX, XLSX, PPTX, HTML).

Skill 02

Video / Audio a texto

Analiza videos con Gemini directo, descarga de YouTube con yt-dlp, o transcribe local con whisper.cpp cuando importa la privacidad.

Skill 03

Plan Paso a Paso

Obliga al sistema a pensar antes de paralelizar. Sin esto, 50 tareas simultáneas colapsan la máquina. Con esto, terminan.

El precio de un plan bien armado es un poquito más de tiempo. El precio de no tenerlo es que se te rompa el proyecto a la mitad.

Todo lo que instala un comando.

El script ./tooling/install.sh es interactivo: te pregunta bloque por bloque qué querés instalar. Si ya tenés algo, lo saltea.

Prerrequisitos

La base del sistema

Homebrew · git · GitHub CLI · Python 3.12 · uv (gestor Python ultra-rápido) · nvm + Node LTS · ffmpeg · Docker Desktop.

Documentos

PDF / Office → Markdown

Video / Audio

Todo formato → texto

Terminal

CLIs modernos

08 · Ritual con el cliente

Del levantamiento al output.

Trabajar con IA sin metodología con el cliente es como manejar F1 sin pits. Este es el flujo que seguimos siempre.

El mantra operativo

MD sucio → MD limpio → Ejecutás

Primero escribimos todo el contexto en un Markdown sucio — notas, requisitos, dudas, lo que sea. Después lo pasamos por un LLM de contexto grande (Gemini, GPT) y lo transformamos en un MD limpio, estructurado, ejecutable. Ese MD limpio es el NV: el prompt final que le pasamos al agente.

La regla del NV: antes de que ejecute, que hable

Antes de comenzar, decime si tenés dudas, qué no te queda claro y qué información te falta. No inicies nada hasta que yo te avise. Cuando estés listo, avanzás por pasos; si algo no avanza, te detenés y me avisás.

Esto resuelve el 80% de los errores antes de que ocurran. La mitad de las veces el agente devuelve preguntas que nos hacen darnos cuenta de que el brief al cliente estaba incompleto. Barato.

Si el MD no avanza, el problema no es el modelo. Es el MD.
09 · Reglas de oro

Lo que no se negocia.

El resumen de lo que, con el tiempo, dejamos de discutir. Si las rompemos, el proyecto lo paga.

01

Conectá GitHub primero.

Repo conectado, permisos de terminal, .env en local. Si no, construyes sobre arena.

02

Un cliente = una carpeta madre.

Incluso tu propia agencia. Así "el contexto de la empresa" siempre está junto.

03

Compilá a los 500k tokens.

No esperes al millón. Compilar temprano es gratis; depurar errores del modelo confundido, no.

04

Reemplazar > duplicar.

Nada de plan_v2_final_final.md. Git guarda el historial.

05

Purgá lo que no se usa.

Carpeta limpia = contexto limpio. El ruido en la carpeta es ruido en el modelo.

06

Todo lo final va a output/.

Cero excepciones. No gastás tiempo buscando entre archivos internos.

07

Un agente, un job.

Cuando un agente hace dos cosas, hace mal las dos. Catorce chicos > tres confundidos.

08

El NV nunca arranca sin confirmar.

"Decime dudas antes de iniciar" es fijo. Te ahorra 40 min de ejecución errada.

09

Plan paso a paso > todo en paralelo.

Paralelización mal hecha colapsa la máquina y quema tokens.

10

Si no avanza, no es el modelo.

Es el prompt, el contexto o el plan. Cambiar de LLM es la última carta.

10 · FAQ

Preguntas frecuentes.

Lo que nos preguntan más seguido. Si no está acá, escribinos por WhatsApp al +51 995 547 575.

¿Esto corre en Windows?
El script tooling/install.sh asume macOS. Para Windows, pedile a Claude/Cursor que aplique lo mismo usando winget o scoop en lugar de brew. Los paquetes Python/Node/Docker tienen instaladores nativos. En Linux, reemplazá brew por apt o dnf.
¿Necesito todas las herramientas?
No. El instalador es interactivo: te pregunta bloque por bloque. Los prerrequisitos sí los recomendamos todos. El resto depende del proyecto — si no vas a procesar video, saltealo.
¿Por qué incluye Gemini y no otros modelos?
El skill video-a-texto aprovecha que Gemini acepta video como input directo (hasta ~1h) — algo que Claude y GPT-4 todavía no ofrecen nativamente así de simple. Conseguís el token gratis en aistudio.google.com/apikey. Para el resto de tareas usá el LLM que prefieras.
¿Qué hace .claude/settings.json?
Define qué comandos puede ejecutar Claude Code sin pedirte autorización cada vez. Pre-aprobamos los seguros (git, brew, pip, npm, etc.) y bloqueamos los peligrosos (rm -rf /, git push --force, lectura de .env). Si querés ser más estricto, editá ese archivo.
¿Es seguro dar tantos permisos preaprobados?
Sí, porque está diseñado con una lista blanca (lo que se permite) y una lista negra crítica (lo que nunca se permite). Operaciones destructivas tipo force-push, reset --hard, lectura de credenciales o borrado masivo siguen pidiendo confirmación explícita.
¿Puedo agregar mis propios agentes?
Sí. Creá un archivo .md en .claude/agents/ con el frontmatter estándar (copiá cualquier agente existente como modelo). Claude Code lo detecta al reiniciar.
¿Cómo actualizo el starter a una versión más nueva? 
git remote add starter https://github.com/tamibot/creators-latam-starter.git
git fetch starter
git merge starter/main --allow-unrelated-histories
Resolvé los conflictos — el agente Versionador te ayuda.
¿Por qué no hay herramienta de web scraping?
La sacamos a propósito para mantener el starter liviano. Si la necesitás para un proyecto específico, agregala en tooling/ del proyecto y documentala ahí.
¿Qué diferencia tiene con los starters de Vercel/Next?
Aquellos son templates de código. Éste es un starter de proceso: reglas, agentes, skills, estructura. Se monta sobre cualquier stack técnico (React, Python, Node, Go...).
Me rompió algo el instalador, ¿qué hago?
El script nunca borra nada y saltea lo que ya tenés. Si algo falló, revisá el output — te dice qué paquete específico no pudo instalar y probablemente tengas que correrlo manualmente (brew install <pkg>) para ver el error real. También podés abrir un issue en el repo.
