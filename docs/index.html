Cómo trabajamos con IA. Metodología primero, herramientas después.

El problema

Instalar un stack no es tener un proceso.

La mayoría de los proyectos con IA fallan no por falta de herramientas — fallan por falta de claridad en qué hay que entregar, por no haber investigado el stack, y por armar el equipo sin plan.

Antes de tocar una sola API: definí el entregable, contrastá el stack, armá el equipo. En ese orden. Siempre.
La metodología

Tres fases. Ninguna se saltea.

Cada fase termina con validación explícita del cliente. No avanzamos hasta que la anterior esté firmada. Al final, tenés un Plan Maestro que el cliente aprueba y recién ahí se ejecuta.

1
FASE 1

Entregables · pensar desde el final

Trabajamos de adelante hacia atrás. Primero definimos qué archivo, pantalla o experiencia tiene que existir cuando terminemos. Después vemos cómo se produce.

¿Qué cosa concreta recibe el cliente cuando el proyecto termina perfecto?

Ejemplo real

Entregable: reporte-leads-abril.xlsx Ubicación: output/reportes/ Frecuencia: semanal, lunes 9am Columnas: · fecha # de Kommo API (lead.created_at) · nombre, email # de Kommo API (contact fields) · empresa # enriquecimiento con Clearbit · canal # de utm_source (HubSpot form) · estado # de Kommo pipeline stage · lifetime_value_usd # calculado: SUM(deals.amount) KPIs al final del reporte: · leads_totales # COUNT(*) · tasa_conversion # qualified / total · CAC # ad_spend / leads_convertidos · LTV:CAC # benchmark > 3:1 Aprobación: El cliente abre el Excel y dice: "sí, esto es lo que necesitaba."
✗ Anti-patrón

"Vamos a integrar la API de Kommo" → eso es stack, no entregable. Volvé al entregable primero.

✓ Validación
  • Formato y ubicación exactos por cada entregable
  • Cada columna/campo con fuente de datos trazable
  • KPIs con fórmula definida
  • Criterio de aprobación documentado
  • Cliente confirma por escrito antes de avanzar
2
FASE 2

Stack Tecnológico · investigar, no adivinar

Con los entregables claros, ahora sí vemos con qué herramientas los producimos. Pero no alcanza con lo que dijo el cliente. Para cada herramienta, research obligatorio.

¿Qué tiene hoy el cliente, qué herramientas elegimos, por qué éstas y no otras?

Fuentes que consultamos (por cada herramienta)

Comunidad

Foros y Reddit

StackOverflow, subreddit específico, HackerNews. ¿Qué problemas reportan los que ya integraron?

Código

GitHub Issues

Issues abiertos de los últimos 6 meses. ¿Hay bugs críticos? ¿Cambios breaking?

Negocio

Precios y limits

Tiers, rate limits, regiones, trampas en letra chica. Proyectar costo mensual real.

Para cada herramienta completamos el template templates/api-research.md → queda documentado en documentation/stack/<herramienta>.md.

También validamos los TDRs del cliente: qué no se puede tocar, qué estándares hay que cumplir (GDPR, ISO), cuánto es el presupuesto, qué deadline es firme.

✓ Validación
  • Research documentado por cada herramienta externa
  • Al menos 2 alternativas evaluadas por decisión
  • Riesgos conocidos y mitigación
  • Costo mensual total estimado y aprobado
  • TDRs del cliente revisados punto por punto

Qué esperar

Timeline realista. Sin sorpresas.

Cuánto tarda cada paso en serio. Si algo dura más de lo indicado, algo está mal — no es normal.

15–30 min

Primera instalación del starter

En una Mac nueva sin nada: descargar Homebrew, Python, ffmpeg, Docker, marker (incluye modelos ~2GB), whisper-cpp, CLIs modernos. Es largo la primera vez, pero no te asustes — la siguiente corrida es en segundos.

2–5 seg

Instalaciones posteriores

El script audita qué tenés instalado y salta lo que ya existe. En un proyecto nuevo clonado, si ya usaste el starter antes, tarda segundos en validar.

30 min – 3 días

Fase 1 del Plan Maestro (Entregables)

Depende de qué tan claro tenga el cliente lo que quiere. Si vino con idea vaga: 3 días de ida y vuelta. Si vino con screenshot del Excel que imagina: 30 minutos.

1–2 días

Fase 2 (Stack + research)

Research obligatorio en foros por cada herramienta externa. Esto no se acelera — es donde evitás los sustos futuros (rate limits, precios, limitaciones no documentadas).

4–8 horas

Fase 3 (Equipo + Gantt)

Con las fases 1 y 2 cerradas, armar el equipo de agentes y el Gantt es rápido. El diagramador-mermaid te hace el Gantt en 15 min.

Variable

Ejecución (post-firma)

Recién acá arranca el código. El Gantt de la Fase 3 ya estableció las semanas. Con el Plan Maestro firmado, es ejecución pura.

¿Sentís que la instalación está colgada? Los modelos de marker tardan la primera vez (~2GB). Si llevás más de 30 minutos sin ver progreso, cancelá con Ctrl+C y correlo de nuevo — el script detecta qué ya bajó.
El sistema que ejecuta

Cuando el Plan Maestro está firmado, el sistema arranca.

Arquitectura de carpetas

Una carpeta madre por cliente. Adentro, una subcarpeta por proyecto. El starter replica esta estructura en todos.

carpeta · cliente · proyecto
📁 Cliente-X/Proyecto-A/
├── CLAUDE.md              # reglas que Claude lee auto
├── .env                   # credenciales locales · NUNCA subir
├── 📁 .claude/
│   ├── settings.json      # permisos preaprobados
│   ├── agents/             # 15 agentes + customs del proyecto
│   └── skills/             # pdf→md, video→texto, plan paso a paso
├── 📁 templates/          # plan-maestro, NV, brief, API research
├── 📁 documentation/      # plan-maestro.md, stack/, bitacora
├── 📁 data_original/      # PDFs, raw — sagrado, no se toca
└── 📁 output/             # entregables finales — lo que el cliente ve

Comandos esenciales

Atajos de Claude Code que sí o sí tenés que usar.

Los comandos que más nos salvan el día. Aprendelos de memoria — son la diferencia entre perder horas o avanzar tranquilo.

crítico
/compact

Compilar el contexto

Cuando la conversación llega cerca de 500k tokens, esto la resume y reinicia. Usalo antes de que el modelo empiece a inventar. Regla: al 50% del contexto, compactás.

crítico
ESC ESC

Rewind (volver atrás)

Doble Escape te deja volver a un turno anterior. Cuando el modelo se equivocó y querés reintentar desde un punto limpio — sin empezar de cero.

/clear

Limpiar el chat

Borra la conversación actual pero mantiene el proyecto abierto. Útil cuando el contexto se ensució y no tiene sentido compactarlo.

/cost

Ver uso de tokens

Muestra cuántos tokens llevás usados en la sesión. Revisá antes de meter otro PDF grande — no quemes el contexto en vano.

/doctor

Health check

Diagnostica tu setup: Node, npm, MCP, permisos. Primer comando cuando algo "raro" pasa. Te ahorra investigar solo.

/agents

Gestionar agentes

Listá, creá, editá agentes desde Claude Code. Útil para crear rápido los agentes custom de la Fase 3 del Plan Maestro.

/init

Generar CLAUDE.md

Si abrís un repo sin CLAUDE.md, este comando te lo arma explorando el código. Base mínima para que el modelo entienda el proyecto.

/config

Settings

Abrir settings del IDE. Ajustar modelo, permisos, tema, idioma. Lo que antes editabas a mano en settings.json.

Atajos de teclado

Shift + Tab

Toggle "plan mode"

Entrás al modo planeamiento antes de ejecutar. El modelo te propone el plan y vos aprobás. Crítico para tareas grandes.

Ctrl + R

Toggle verbose / modo extendido

Ver más detalle de lo que hace el modelo. Útil cuando algo falla y querés saber exactamente qué pasó.

Ctrl + C

Cancelar ejecución

Para el comando que está corriendo. No cierra la sesión. Si el agente se está yendo por la tangente, cortalo.

Mejores prácticas

Tips que aprendimos a la mala.

Reglas pequeñas pero firmes que, con el tiempo, dejamos de cuestionar. Son la diferencia entre un proyecto que avanza y uno que se empantana.

Formatos y naming

📄

Markdown por default, para todo lo que Claude tenga que leer

Notas, briefs, planes, transcripciones. Los modelos leen MD 10× más rápido que PDFs, DOCX o HTML. Si entra un PDF a data_original/, pasalo a MD con marker antes de que el agente lo toque.

🏷️

Archivos con nombre descriptivo, sin espacios ni acentos

Usá reporte-leads-abril2026.xlsx, no "Reporte Leads (Abril 2026).xlsx". Los espacios rompen comandos, los acentos rompen encoding. kebab-case siempre.

📂

Un cliente = una carpeta madre. Siempre.

Incluso tu propia agencia. Así cuando decís "el contexto de la empresa" ya está todo junto. El modelo aterriza en segundos.

Cómo hablarle al modelo

🎯

Todo mensaje largo arranca con "Antes de comenzar, dime si tienes dudas"

El prompt del NV te ahorra 40 min de ejecución errada. Template completo en templates/nv-prompt.md. Si no usás esto, el modelo adivina — y adivinar en producción es caro.

🧭

Instrucciones claras > instrucciones extensas

No escribas 2 pantallas de prompt. Escribí 5 bullets claros con: objetivo, entrada, salida esperada, restricciones, criterio de corte. Si necesitás más, partí la tarea en 2.

🛑

Si el modelo se va por la tangente, cortá con Ctrl+C

No dejes que se auto-corrija 5 veces quemando tokens. Cortá, reformulá el prompt, arrancá de nuevo. Es más barato.

Higiene del contexto

♻️

Compilá a los 500k tokens

No esperes al millón. /compact temprano es gratis; depurar errores del modelo confundido, no. El síntoma de "hay que compilar" es el modelo olvidándose de archivos que hace rato usaste.

🧹

Purgá la carpeta cada ciertos pasos

El agente purgador barre archivos no usados, TODOs viejos, carpetas vacías. Carpeta limpia = contexto limpio. El ruido en la carpeta es ruido en el modelo.

🚫

Reemplazar > duplicar. Confiá en git.

Nada de plan_v2_final_FINAL.md. El versionador edita en sitio y git guarda el historial. Si necesitás revertir, hacé git log.

Seguridad

🔐

Credenciales: sólo en .env, nunca en código

El credentials-manager chequea antes de cada commit. Si querés estar más seguro, usá gitleaks en un pre-commit hook. Una credencial en un commit público = rotarla YA.

⛔

Si no usás Claude Code, no setées ANTHROPIC_API_KEY

Claude Code usa su propia sesión. Solo setées la key si hacés llamadas a la API desde código. Menos keys en el .env, menos superficie de ataque.

👀

Antes de cada commit grande: /agents → guardian-reglas

Audita que las 10 reglas de oro se cumplan. Detecta duplicados, credenciales filtradas, carpetas muertas. Si reporta crítico, no commites hasta resolver.

Metodología
¿Por qué arrancan con "Plan Maestro" y no directamente con el código?
Porque sin entregable definido, stack validado y equipo armado, el código que escribís es al pedo. El 80% de los proyectos fallidos que vimos empezaron con "dale, codeá". El Plan Maestro toma 2–5 días y ahorra semanas.
¿Qué pasa si el cliente no quiere pasar por las 3 fases?
Le explicamos que no trabajamos sin plan firmado. Si insiste, agradecemos pero no tomamos el proyecto. Los proyectos sin plan terminan siendo los más problemáticos y no son rentables para ninguno de los dos.
¿Es obligatorio investigar en foros en la Fase 2?
Sí. Las docs oficiales siempre están incompletas o desactualizadas. Los foros te muestran los problemas reales de la gente que ya integró. Para cada API, revisamos StackOverflow, Reddit, GitHub issues recientes y HackerNews. Queda documentado en documentation/stack/<herramienta>.md.
¿Puedo agregar una Fase 4 (deployment / operaciones)?
Sí — las 3 fases son el mínimo. Si el proyecto lo amerita, el onboarding-pm puede sumar fases específicas (deployment, monitoring, handover al equipo del cliente). El template es extensible.
Setup e instalación
¿Cuánto tarda instalar todo?
Primera vez en una Mac vacía: 15–30 min (descarga Homebrew + Python + modelos marker ~2GB). En corridas siguientes: segundos — el instalador detecta qué ya tenés y salta lo instalado. No reinstala nada.
Necesito ANTHROPIC_API_KEY?
Si usás Claude Code como IDE, no — Claude Code usa su propia sesión de login. Solo necesitás ANTHROPIC_API_KEY si vas a llamar a la API de Anthropic desde tu código (ej: un script Python). El .env.example del starter lo aclara.
¿Corre en Windows?
El script asume macOS. Para Windows, pedile a Claude/Cursor que aplique lo mismo usando winget o scoop. Las herramientas (Python, Node, Docker, yt-dlp, marker) tienen soporte Windows nativo. Linux: reemplazar brew por apt/dnf.
¿Qué hace .claude/settings.json?
Define qué comandos puede ejecutar Claude Code sin pedirte autorización cada vez. Pre-aprobamos los seguros (git, brew, pip, npm) y bloqueamos los peligrosos (rm -rf /, git push --force, leer .env).
¿Puedo usar Cursor o Codex en vez de Claude Code?
Sí. La estructura de .claude/agents/ y .claude/skills/ es específica de Claude Code, pero las reglas en CLAUDE.md y los templates son universales — cualquier LLM puede seguirlos. Adaptá agentes en formato del IDE que uses.
Agentes y ejecución
¿Puedo armar agentes custom para mi proyecto?
Sí — el onboarding-pm los propone en la Fase 3 del Plan Maestro. Creás un .md en .claude/agents/ con frontmatter estándar (copiá cualquier agente existente como modelo). Claude Code lo detecta al reiniciar.
¿Cuánto cuesta en tokens una sesión típica?
Depende del proyecto. Una sesión de Fase 1 (entregables) suele ser ~20k-50k tokens input. Una sesión de Fase 3 con generación de código: 100k-500k. Usá /cost para ver en tiempo real. El suscription de Claude Pro suele cubrirlo bien.
¿Qué hago si un agente se va por la tangente?
1) Ctrl+C para cortar. 2) ESC ESC para volver al turno anterior (rewind). 3) Reformulá el prompt siguiendo el formato del NV (objetivo, entrada, salida, restricciones, corte). Si el agente se va por la tangente seguido, el prompt probablemente es ambiguo — pediles al arquitecto que lo revise.
¿Cómo actualizo el starter cuando sacamos una versión nueva? 
git remote add starter https://github.com/tamibot/creators-latam-starter.git
git fetch starter
git merge starter/main --allow-unrelated-histories
El agente versionador te ayuda con los conflictos.
¿Qué diferencia tiene con starters de Vercel / Next?
Aquellos son templates de código. Éste es un starter de proceso: reglas, agentes, metodología de 3 fases. Se monta sobre cualquier stack técnico (React, Python, Node, Go).
Troubleshooting
La instalación se colgó — ¿qué hago?
Los modelos de marker tardan la primera vez (~2GB). Si llevás más de 30 min sin ver progreso, cancelá con Ctrl+C y correlo de nuevo — el script detecta qué ya bajó y retoma. Si insiste, instalá ese paquete manual: uv pip install marker-pdf.
El modelo no encuentra un archivo que yo veo en la carpeta
Probablemente es un problema de contexto. Reiniciá con /clear o /compact y volvé a mencionar el archivo con ruta completa. Si persiste, es que el archivo está fuera del workspace del modelo — chequeá los permisos en .claude/settings.json.
Docker Desktop no arranca después de instalar
Abrilo manualmente desde Applications al menos una vez — pide permisos de macOS que no se pueden dar automáticamente. Después ya arranca solo.
"command not found: brew" después de instalarlo
Agregá Homebrew al PATH. En Apple Silicon: 
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zshrc
source ~/.zshrc
"externally-managed-environment" al instalar con pip
Es Python 3.12 protegiéndote de romper el sistema. Usá uv pip install --system (incluido en el starter) o pip3 install --user --break-system-packages si sabés lo que hacés.
El agente no aparece en /agents
Cerrá y reabrí Claude Code. Los agentes se cargan al iniciar la sesión. Si aún no aparecen, chequeá que el frontmatter del archivo esté bien (tiene que tener ---, name:, description:, otro ---).
